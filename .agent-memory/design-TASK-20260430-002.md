## Problem Statement

After merging PR #86 (TASK-20260430-001), a post-merge developer simulation revealed 5 DX gaps and
1 missing test that would have caught them:

1. `kindling run` crashes because `WatermarkEntityFinder` is abstract with no concrete binding in
   standalone mode. The DI chain is: `DataPipesExecution → (EntityReadPersistStrategy) →
   SimpleReadPersistStrategy → WatermarkService → WatermarkManager → WatermarkEntityFinder`.
   `WatermarkManager` is auto-bound as a singleton, but `WatermarkEntityFinder` has no binding.

2. 86 `print()` calls in `bootstrap.py` and 23 in `spark_config.py` flood stdout on every
   `kindling run` invocation, violating the Conventions "no bare print() in library code" rule
   and making CLI output unusable.

3. `kindling validate` was added in PR #86 but lacks the `--env` option that `kindling run`
   already has. Running `kindling validate --env local` returns "No such option: --env".

4. The `kindling new` next-steps message says "cp .env.example .env  # fill in your credentials"
   with no indication that credentials are optional for local dev (`--env local`).

5. `spark-kindling-cli` is commented out in the scaffold `pyproject.toml.j2`, so after
   `poetry install` the `kindling` command is not on PATH.

6. No component test exercises the real `initialize_framework` + real `GlobalInjector.get(DataPipesExecution)`
   end-to-end. Any DI binding regression (like item 1 above) is invisible to the test suite.

## Item 1: WatermarkEntityFinder — Decision + Pseudocode

### Decision: Option C — standalone platform auto-registers a NullWatermarkEntityFinder during initialize_framework

**Rationale:**

- Option B (scaffold `app.py.j2`) requires every new developer to know about `WatermarkEntityFinder`
  before they've read a line of the docs. The `eventhub-provider-test-app/main.py` shows this
  pattern already causes copy-paste boilerplate. It burdens developers and will be forgotten.
- Option A (CLI injects before `run_datapipes`) only fixes `kindling run` — programmatic use
  and tests that call `initialize_framework(platform="standalone")` directly would still fail.
- Option C places the binding at the exact point where the standalone platform is declared
  "ready to use". `initialize_framework` already calls `initialize_platform_services` which
  resolves the platform. Immediately after that call, we can conditionally bind the null
  implementation. This keeps production code unaffected (production apps bind a real
  `WatermarkEntityFinder` in their `app.py`) and makes standalone mode work out of the box.

**Production safety:** The null implementation raises a clear `NotImplementedError` with an
actionable message if `WatermarkManager` is actually called (e.g., when `use_watermark=True`
on a pipe). It does NOT silently swallow errors — it fails loudly but only when the feature
is actually used. Developers who need real watermarking must bind their own finder, just as
they do today in `eventhub-provider-test-app/main.py`. The convention check test (Item 6) will
catch future regressions.

### File: `packages/kindling/watermarking.py`

Add after the `WatermarkEntityFinder` ABC definition (after line 37):

```python
class NullWatermarkEntityFinder(WatermarkEntityFinder):
    """No-op WatermarkEntityFinder for standalone/local-dev mode.

    Bound automatically when platform="standalone". Raises NotImplementedError
    if watermarking is actually invoked — use WatermarkEntityFinder to provide
    a real implementation when pipes use use_watermark=True.
    """

    def get_watermark_entity_for_entity(self, context: str):
        raise NotImplementedError(
            "WatermarkEntityFinder is not configured for standalone mode. "
            "Bind a WatermarkEntityFinder implementation before using watermarks: "
            "GlobalInjector.bind(WatermarkEntityFinder, MyWatermarkEntityFinder)"
        )

    def get_watermark_entity_for_layer(self, layer: str):
        raise NotImplementedError(
            "WatermarkEntityFinder is not configured for standalone mode. "
            "Bind a WatermarkEntityFinder implementation before using watermarks: "
            "GlobalInjector.bind(WatermarkEntityFinder, MyWatermarkEntityFinder)"
        )
```

### File: `packages/kindling/bootstrap.py`

In `initialize_framework`, immediately after `initialize_platform_services(...)` returns
(currently line 1463), add a conditional bind:

```python
platformservice = initialize_platform_services(platform, config_service, logger)
logger.info("Platform services initialized")

# [planner] bind NullWatermarkEntityFinder for standalone so DI resolves — TASK-20260430-002
if platform == "standalone":
    from kindling.injection import GlobalInjector
    from kindling.watermarking import NullWatermarkEntityFinder, WatermarkEntityFinder

    # Only bind if no concrete binding exists yet (app.py may have already bound one)
    try:
        existing = GlobalInjector.get(WatermarkEntityFinder)
        if isinstance(existing, NullWatermarkEntityFinder):
            pass  # already the null impl; nothing to do
    except Exception:
        GlobalInjector.bind(WatermarkEntityFinder, NullWatermarkEntityFinder)
        logger.debug("Bound NullWatermarkEntityFinder for standalone mode")
```

**Note on check-then-bind:** The injector's `auto_bind=True` will raise on `get(abstract)` only
if there is truly no binding. If the user's `app.py` called `initialize_framework` and then
bound their own finder, the try/except preserves it. If the injector resolves to a concrete
class (including `NullWatermarkEntityFinder`), we leave it alone.

A simpler alternative is to bind unconditionally before the user's `app.py` initialize runs
(i.e., register as a default in `platform_standalone.py`'s `create_platform_service`). However,
because `initialize_framework` is called from within `app.py.initialize()`, and `app.py` can
call `GlobalInjector.bind(WatermarkEntityFinder, MyFinder)` either before or after
`initialize_framework`, the safest approach is:

1. After `initialize_platform_services` returns.
2. Only if `WatermarkEntityFinder` does not already resolve to a non-abstract binding.

The injector `auto_bind=True` will attempt to instantiate abstract classes and raise. Use the
presence/absence of an exception as the "is it bound?" signal:

```python
# [planner] standalone null-bind guard — TASK-20260430-002
if platform == "standalone":
    from kindling.watermarking import NullWatermarkEntityFinder, WatermarkEntityFinder
    _wef_already_bound = False
    try:
        candidate = GlobalInjector.get_injector().binder._bindings.get(WatermarkEntityFinder)
        _wef_already_bound = candidate is not None
    except Exception:
        _wef_already_bound = False

    if not _wef_already_bound:
        GlobalInjector.bind(WatermarkEntityFinder, NullWatermarkEntityFinder)
        logger.debug("Bound NullWatermarkEntityFinder for standalone mode")
```

This accesses the internal `binder._bindings` dict (injector library implementation detail).
If it becomes fragile, fall back to the try/get approach. The implementer should pick whichever
is more resilient to injector version changes — both are valid.

## Item 2: Debug print suppression — Decision + Strategy

### Counts
- `bootstrap.py`: **86** `print()` calls
- `spark_config.py`: **23** `print()` calls
- **Total: 109** bare prints in library code.

### Decision: Remove debug/emoji prints entirely; route operational prints through the logger; no KINDLING_QUIET env var needed

**Rationale:** The `KINDLING_QUIET=1` escape hatch is unnecessary if the prints are removed or
routed to the existing logger. The key constraint is that some prints fire before `logger` is
available (before `configure_injector_with_config` returns). Analysis of the code shows three
categories:

#### Category A — Pre-logger "we're starting" announcements (remove entirely)

These fire before the logger is created and duplicate information that the logger will emit:

| File | Line | Content |
|------|------|---------|
| `spark_config.py` | 124 | `print(f"Initializing DynaconfConfig:")` |
| `spark_config.py` | 148 | `print(f"✓ DynaconfConfig initialized")` |
| `spark_config.py` | 507 | `print("Setting up config ...")` |
| `spark_config.py` | 539 | `print("✓ Config configured in GlobalInjector")` |
| `bootstrap.py` | 1249 | `print(f"initialize_framework: initial_config = {config}")` |
| `bootstrap.py` | 1253 | `print("Framework already initialized, skipping re-initialization")` |

**Action: Delete these lines entirely.** They fire before any logger exists and convey information
already present in `logger.info("Starting framework initialization")` (line 1339) and subsequent
log lines.

#### Category B — DEBUG-level diagnostic prints (route to logger.debug or logger.info)

These fire mid-initialization after `logger` is available. They have legitimate diagnostic value
for troubleshooting but must not appear in normal CLI stdout:

| File | Lines | Content |
|------|-------|---------|
| `spark_config.py` | 154–165 | Dynaconf key dump (DEBUG block) |
| `spark_config.py` | 184 | Reverse-translation lines |
| `spark_config.py` | 350, 357, 379, 390, 392, 408, 416, 423 | Reload lifecycle events |
| `spark_config.py` | 528 | workspace_id error |
| `spark_config.py` | 633, 638, 673 | Platform detection |
| `bootstrap.py` | 109 | SparkConf merge count |
| `bootstrap.py` | 302–331 | Artifacts path setup |
| `bootstrap.py` | 379–407 | Config file download |
| `bootstrap.py` | 528 | workspace_id warning |
| `bootstrap.py` | 788–840 | Package install progress |
| `bootstrap.py` | 862–1092 | Extension install progress |
| `bootstrap.py` | 1097–1137 | Required packages/extensions loading |
| `bootstrap.py` | 1278, 1287 | Platform detection warnings |
| `bootstrap.py` | 1312 | Config service already initialized |
| `bootstrap.py` | 1408–1420 | Full kindling config debug dump |
| `bootstrap.py` | 1479–1506 | load_local debug |

**Action: Replace `print(...)` with `logger.debug(...)` (or `logger.info(...)` for lifecycle
milestones, `logger.warning(...)` for warnings, `logger.error(...)` for errors).**

For the pre-logger section (before `logger` is obtained at line 1338), use Python's built-in
`logging.getLogger("kindling.bootstrap").debug(...)` as a fallback. Most pre-logger debug
prints in the 1249–1312 range can simply be deleted as they repeat state available in the config
object. The remainder (platform detection warnings at 1278/1287) should use `logging.getLogger`.

#### Category C — Completely leftover debug (remove entirely)

These are clearly ad-hoc debug prints that should never have been committed:

| File | Lines | Content |
|------|-------|---------|
| `spark_config.py` | 154–165 | `🔍 DEBUG: All Dynaconf keys loaded from YAML:` dump |
| `bootstrap.py` | 1408–1417 | `🔍 DEBUG: Full kindling config section:` dump |
| `bootstrap.py` | 1479–1493 | `🔍 DEBUG: kindling.bootstrap.load_local = ...` dumps |
| `bootstrap.py` | 1513–1525 | `🚀 ATTEMPTING TO RUN APP: ...` / `✅ Got DataAppRunner` / etc. |

**Action: Delete these lines entirely.** The emoji prefix and "DEBUG:" prefix indicate they
were temporary investigative prints added during development.

#### Category D — `spark_config.py` `print()`-based logger (special case)

Lines 597–606 in `spark_config.py` contain a `print()`-based logger implementation used as a
fallback. This is intentional design (logging before logging is configured) but should be
renamed/isolated as `_bootstrap_print_logger` and documented, not removed.

**Action: Keep as-is; add a comment explaining the intent.**

### Implementation order

1. Delete Category C lines first (no risk of breaking anything).
2. Delete Category A lines.
3. Replace Category B `print()` calls with `logger.debug/info/warning/error(...)`.
4. For the handful of pre-logger prints that remain (lines before `logger =` assignment),
   route to `logging.getLogger("kindling.bootstrap").debug(...)`.
5. Document Category D.

### No KINDLING_QUIET needed

All 109 prints fall into A, B, or C. After cleanup, no print remains in library code that
a user would ever need to suppress via an env var. The logger is already configurable via
`kindling.telemetry.logging.level` in `settings.yaml`.

## Item 3: `kindling validate` missing `--env` — Exact change

**File:** `/workspaces/kindling/packages/kindling_cli/kindling_cli/cli.py`

The `run_pipe` command (line 436) has this decorator and signature:
```python
@cli.command("run")
@click.argument("pipe_id")
@click.option("--env", default=None, help="Environment overlay (default: KINDLING_ENV or 'local')")
@click.option("--app", ...)
@click.option("--config", ...)
def run_pipe(pipe_id: str, env: Optional[str], app_path: Optional[Path], config_dir: Optional[Path]) -> None:
```

The `validate_app` command (line 488) currently has:
```python
@cli.command("validate")
@click.option("--app", "app_path", ...)
def validate_app(app_path: Optional[Path]) -> None:
    ...
    _load_app_module(resolved_app, env="local")  # hardcoded "local"
```

**Exact change:**

Add `--env` option decorator and pass it through to `_load_app_module`:

```python
@cli.command("validate")
@click.option(
    "--env",
    default=None,
    help="Environment overlay (default: 'local')",
)
@click.option(
    "--app",
    "app_path",
    default=None,
    type=click.Path(path_type=Path, dir_okay=False, exists=False),
    help="Path to app.py",
)
def validate_app(env: Optional[str], app_path: Optional[Path]) -> None:
    ...
    resolved_env = env or "local"
    _load_app_module(resolved_app, env=resolved_env)
```

The decorator order must put `--env` before `--app` (matching the `run` command pattern). The
function signature gains `env: Optional[str]` as a parameter before `app_path`.

## Item 6: DI wiring component test — Pseudocode

### Placement

**File:** `tests/unit/test_di_wiring_standalone.py`

The pyproject.toml.j2 template shows `test-component = "pytest tests/component -v"` which
corresponds to a `tests/component/` directory in scaffolded projects. However, the Kindling
_framework_ repo itself has `tests/unit/` with no `tests/component/`. Given that:

- CONVENTIONS.md says: "Tests mirror source: `tests/unit/`, `tests/integration/`, `tests/system/`"
- The test uses real `initialize_framework` (no Spark, standalone platform) — this is equivalent
  to what the framework calls a "unit" test (no cloud infra), just exercising real DI
- Existing DX tests live in `tests/unit/test_cli_local_dev_dx.py`

Placement: **`tests/unit/test_di_wiring_standalone.py`**

The test uses `poe test-unit` (not integration), consistent with standalone platform requiring
no cloud resources.

### Pseudocode

```python
# [tester] DI wiring component test for standalone platform — TASK-20260430-002
"""Component test: real initialize_framework + real GlobalInjector.get(DataPipesExecution).

No mocking. Verifies that the DI graph resolves end-to-end after initialize_framework()
is called with platform="standalone". Any missing binding (e.g., WatermarkEntityFinder)
causes this test to fail before it reaches production.
"""
import pytest
from kindling.bootstrap import initialize_framework
from kindling.data_pipes import DataPipesExecution, DataPipesRegistry
from kindling.data_entities import DataEntityRegistry
from kindling.injection import GlobalInjector


@pytest.fixture(autouse=True)
def reset_kindling():
    """Reset DI state before and after each test to prevent cross-test pollution."""
    from kindling.data_entities import DataEntities
    from kindling.data_pipes import DataPipes

    DataEntities.reset()
    DataPipes.reset()
    GlobalInjector.reset()
    yield
    DataEntities.reset()
    DataPipes.reset()
    GlobalInjector.reset()


def test_initialize_framework_standalone_resolves_datapipes_execution():
    """initialize_framework(standalone) must produce a resolvable DataPipesExecution."""
    initialize_framework({"platform": "standalone", "environment": "local"})

    executer = GlobalInjector.get(DataPipesExecution)
    assert executer is not None


def test_initialize_framework_standalone_resolves_entity_registry():
    """DataEntityRegistry must resolve after standalone init."""
    initialize_framework({"platform": "standalone", "environment": "local"})

    registry = GlobalInjector.get(DataEntityRegistry)
    assert registry is not None


def test_initialize_framework_standalone_resolves_pipe_registry():
    """DataPipesRegistry must resolve after standalone init."""
    initialize_framework({"platform": "standalone", "environment": "local"})

    registry = GlobalInjector.get(DataPipesRegistry)
    assert registry is not None


def test_datapipes_execution_run_datapipes_with_no_pipes_does_not_raise():
    """DataPipesExecution.run_datapipes([]) must not raise after standalone init.

    This is the minimal smoke test: DI is fully wired and execution is callable.
    No entities or pipes are registered, so it should complete immediately.
    """
    initialize_framework({"platform": "standalone", "environment": "local"})

    executer = GlobalInjector.get(DataPipesExecution)
    # An empty list means no pipes are executed — but the DI chain must fully resolve.
    executer.run_datapipes([])


def test_null_watermark_entity_finder_raises_on_use():
    """NullWatermarkEntityFinder must raise NotImplementedError, not silently swallow it.

    This protects against the null impl masking watermark bugs.
    """
    from kindling.watermarking import NullWatermarkEntityFinder
    from kindling.injection import GlobalInjector
    from kindling.watermarking import WatermarkEntityFinder

    initialize_framework({"platform": "standalone", "environment": "local"})

    finder = GlobalInjector.get(WatermarkEntityFinder)
    assert isinstance(finder, NullWatermarkEntityFinder)

    with pytest.raises(NotImplementedError, match="WatermarkEntityFinder is not configured"):
        finder.get_watermark_entity_for_entity("some.entity")


def test_custom_watermark_entity_finder_not_overwritten():
    """A WatermarkEntityFinder bound before initialize_framework must not be replaced.

    This ensures apps that bind their own finder keep it after framework init.
    """
    from kindling.watermarking import WatermarkEntityFinder

    class MyFinder(WatermarkEntityFinder):
        def get_watermark_entity_for_entity(self, context):
            return None
        def get_watermark_entity_for_layer(self, layer):
            return None

    GlobalInjector.bind(WatermarkEntityFinder, MyFinder)
    initialize_framework({"platform": "standalone", "environment": "local"})

    finder = GlobalInjector.get(WatermarkEntityFinder)
    assert isinstance(finder, MyFinder), (
        "initialize_framework must not replace a user-provided WatermarkEntityFinder binding"
    )
```

## Test Strategy

### Existing tests
All 1106 existing unit tests must pass. The changes to `bootstrap.py` and `spark_config.py`
remove/route `print()` calls and add a null WEF binding — neither change alters any logic path
that existing tests exercise. Run `poe test-unit` to verify.

### New tests (Item 6)
The 6 new test functions in `test_di_wiring_standalone.py` cover:
- `DataPipesExecution` resolves (the original bug)
- `DataEntityRegistry` and `DataPipesRegistry` resolve
- `run_datapipes([])` executes end-to-end without raising
- `NullWatermarkEntityFinder` raises `NotImplementedError` on use (not silent)
- User-provided `WatermarkEntityFinder` binding is preserved by `initialize_framework`

### Existing validate test
The existing `test_validate_good_registry_checks_pass` in `test_cli_local_dev_dx.py` currently
passes `--app` but not `--env`. After Item 3, `--env local` should be added to the test invocation
to verify the new option is accepted and forwarded.

Add to `test_cli_local_dev_dx.py`:
```python
def test_validate_accepts_env_option(monkeypatch):
    """--env option must be accepted by `kindling validate`."""
    # ... setup same as test_validate_good_registry_checks_pass ...
    result = runner.invoke(cli, ["validate", "--env", "local", "--app", str(app_path)])
    assert result.exit_code == 0
    assert "No such option" not in result.output
```

### Print-suppression smoke test
No dedicated test needed for the print removal. The `poe test-unit` run itself will catch
any import errors introduced by the cleanup. The CI integration test exercises `initialize_framework`
end-to-end and its output is validated — if any critical print was load-bearing (it shouldn't be),
the integration test would fail.
