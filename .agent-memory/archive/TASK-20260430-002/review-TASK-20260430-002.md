## Findings

### MINOR — NullWatermarkEntityFinder error message diverges from design spec
**File:** `packages/kindling/watermarking.py:44`
**Issue:** The design spec specifies the error message should read:
> `"WatermarkEntityFinder is not configured for standalone mode. Bind a WatermarkEntityFinder implementation before using watermarks: GlobalInjector.bind(WatermarkEntityFinder, MyWatermarkEntityFinder)"`

The implemented message reads:
> `"No WatermarkEntityFinder is configured. Bind a concrete WatermarkEntityFinder before using watermark-enabled reads or writes."`

The implemented message omits the actionable `GlobalInjector.bind(...)` example and the phrase "for standalone mode" which tells the developer *why* they hit this error. The test in `test_di_wiring_standalone.py` asserts `match="WatermarkEntityFinder is not configured"` which passes under both phrasings, so the divergence is not caught by tests.

**Suggestion:** Update the message to include the `GlobalInjector.bind(WatermarkEntityFinder, MyWatermarkEntityFinder)` example from the spec. Not a blocker — the error still fails loudly and is actionable — but the spec message is more helpful to a first-time developer.

### MINOR — `--env` decorator order is reversed from design spec (cosmetic, not functional)
**File:** `packages/kindling_cli/kindling_cli/cli.py:488-496`
**Issue:** The design spec says: *"The decorator order must put `--env` before `--app` (matching the `run` command pattern)."* The `run` command decorates `--env` first (line 438), then `--app` (line 443). In the implemented `validate` command, `--app` is decorated first (line 489) and `--env` second (line 496). Click processes decorators bottom-up, so the function parameter order (`env, app_path`) is still correct and the CLI works identically. This is purely cosmetic/convention.

**Suggestion:** Reorder decorators so `--env` appears before `--app`, matching the `run` command and the spec. No functional impact, but improves consistency for readers comparing the two commands.

### MINOR — DI test uses subprocess isolation instead of in-process `initialize_framework`
**File:** `tests/unit/test_di_wiring_standalone.py`
**Issue:** The design spec pseudocode shows in-process tests using `initialize_framework` directly with a `reset_kindling` autouse fixture. The implementation instead uses a `_run_fresh_python` helper that spawns a subprocess for each test case. This is a valid approach (avoids cross-test DI state pollution without a reset fixture) and the tests are marked `@pytest.mark.requires_spark` so they skip in restricted sandboxes. However:
1. The subprocess approach means test failures show up as stdout/stderr strings rather than Python tracebacks, reducing debuggability.
2. The design spec's `test_null_watermark_entity_finder_raises_on_use` test (which calls `pytest.raises`) is not present; instead a bare `try/except` is used inside the subprocess string. If the `NotImplementedError` is NOT raised, the subprocess exits with code 1 due to `raise AssertionError(...)` which is caught by `assert result.returncode == 0` — functional but less idiomatic.
3. The design also specified tests for `DataEntityRegistry` and `DataPipesRegistry` resolving independently; the implementation combines all checks into two test functions rather than five discrete ones. Coverage is equivalent.

**Suggestion:** Not a blocker. The subprocess approach is pragmatically sound for avoiding DI state isolation problems. Consider adding a brief comment explaining *why* subprocess isolation was chosen over in-process reset.
