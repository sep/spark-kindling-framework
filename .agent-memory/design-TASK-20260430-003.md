## Constraints

- CONVENTIONS.md: no bare `print()` in library code — use structured logging
- CONVENTIONS.md: `ConfigService` is an ABC with `get(self, key, default=None)` — concrete override may use sentinel default; ABC default is documentation, not enforced
- `spark_session.py` comment: "Can't use DI at this point as this is needed before DI is available" — must use `logging.getLogger(__name__)` directly, not DI-based logger
- `notebook_framework.py` has no module-level logger — need to add one
- `data_apps.py` has `self.logger` (from `spark_log_provider`) — use it
- Template changes affect generated scaffold output; verify j2 syntax is valid
- All existing 1106 unit tests must continue to pass after changes

## Options Considered

### G9 — Log level for missing key

**Option A — `logger.warning`**
Pros: Visible by default; catches misconfigured keys early.
Cons: `ConfigService.get()` is designed to return None for optional keys — most callers intentionally call `config.get("some.optional.key")` expecting None. Warning on every such call would be extremely noisy (hundreds per bootstrap).

**Option B — `logger.debug`** ← RECOMMENDED
Pros: Zero noise at INFO level; available to developers actively debugging. Preserves the intentional "optional get" semantics of the API.
Cons: Less visible to developers who haven't turned up log level.

**Option C — No sentinel, no log**
Pros: No change.
Cons: Misses G9 entirely.

**Decision:** Option B — `logger.debug` with sentinel.

### G12 — Which test fixture to use for Kindling execution test

**Option A — Use existing `_setup` (spark_abfss)**
Cons: Requires Azure credentials; defeats the point of testing local execution. Most local developers won't run these tests.

**Option B — New fixture using `spark_local` + standalone init** ← RECOMMENDED
Pros: Works without Azure credentials; memory provider entities execute without ABFSS. Tests the real Kindling execution path.
Cons: `initialize(env="local")` is a global side effect — needs isolation via `reset_kindling`.

**Option C — Component test (not integration)**
Could live in `tests/component/` per CONVENTIONS.md. However, the scaffold already has an integration test template; adding a Kindling-execution test there makes it discoverable. Reserve component tests for DI-wiring.

**Decision:** Option B — new class in integration template using `spark_local` + standalone init, with `reset_kindling` for isolation.

### G5 — README quick start placement

**Option A — Replace existing Quickstart with CLI section only**
Cons: Removes the notebook-oriented snippet that cloud platform users rely on.

**Option B — Add "CLI Quick Start" before existing Quickstart** ← RECOMMENDED
Pros: Both paths covered; new section is the first thing a developer sees after Install. Rename existing section "Notebook Quick Start" for clarity.

### G8 — `data_apps.py` categorisation

Two distinct groups:

**Group 1 — `run_app()` operational prints (lines 450–516): route to `self.logger`**
These are legitimate step-tracking; `self.logger` is available throughout `run_app()`:
- `🚀 DataAppManager.run_app() starting` → `self.logger.info("run_app starting: %s", app_name)`
- `📋 Step N: ...` → `self.logger.info("Step N: ...", ...)`
- `✅ Step N COMPLETE: ...` → `self.logger.info("Step N complete: ...", ...)`
- `   - Entry point: ...` / `   - PyPI deps: ...` / `   - Lake reqs: ...` → `self.logger.debug(...)`
- `🎉 SUCCESS: App completed` → `self.logger.info("App completed: %s", app_name)`
- `❌ FAILED: App failed` → `self.logger.error("App failed: %s: %s", app_name, e)` (traceback goes to `self.logger.exception`)
- `📋 Full traceback:` + `traceback.print_exc()` → DELETE (replace with `self.logger.exception(...)`)
- `🧹 Cleaning up temp dir` → `self.logger.debug("Cleaning up temp dir: %s", temp_dir)`

**Group 2 — `_execute_app()` debug prints (lines 1259–1304): DELETE ENTIRELY**
All explicitly labeled DEBUG or exposing internal internals (code content, dict sizes). The method already has `self.logger.info` / `self.logger.debug` / `self.logger.error` calls that cover the important state. The debug prints duplicate or exceed what should be logged at any level.
- `🔍 DEBUG: Starting _execute_app` → DELETE
- `Code length: ...` / `First 200 chars of code: ...` → DELETE (sensitive content)
- `✅ Created exec_globals ...` → DELETE
- `✅ Updated exec_globals ...` → DELETE
- `🔨 Compiling code ...` → DELETE
- `✅ Code compiled successfully` → DELETE
- `▶️ Executing compiled code...` → DELETE
- `✅ Code execution completed normally` → DELETE
- `✅ App called sys.exit(...)` → DELETE (already raises or passes)
- `📊 Execution result: ...` → DELETE
- `❌ Exception in _execute_app` + `📋 Traceback:` + `traceback.print_exc()` → DELETE (already has `self.logger.error` below)

### G9 — `DynaconfConfig.get()` sentinel

In `spark_config.py`, add at module level:
```python
_MISSING = object()
```

Change `DynaconfConfig.get()`:
```python
def get(self, key: str, default: Any = _MISSING) -> Any:
    with self._config_lock:
        if self.spark:
            try:
                spark_value = self.spark.conf.get(key.upper())
                if spark_value is not None:
                    return spark_value
            except Exception:
                pass

        result = self.dynaconf.get(key, None)
        if result is None and default is _MISSING:
            _CONFIG_LOGGER.debug("Config key not found (no default supplied): %s", key)
            return None
        return result if result is not None else (None if default is _MISSING else default)
```

Note: the `ConfigService` ABC declares `default: Any = None`. The concrete class overrides with `_MISSING`. Python does not enforce default values in overrides — this is safe.

Also update `get_fresh()` similarly if it has the same pattern.
