# Release Notes - v0.9.0

Version 0.9.0 renames the PyPI distribution to **spark-kindling** and collapses the three per-platform runtime wheels into a single combined wheel with platform extras. Platform service discovery is now driven by `importlib.metadata` entry points, making the loader extensible to third-party platforms without modifying core.

The Python import name is unchanged: notebooks and callers continue to `import kindling.*` exactly as before.

## What's Changed

### Distribution rename

- Runtime distribution renamed: `kindling` → `spark-kindling`.
- Design-time distributions renamed: `kindling-cli` → `spark-kindling-cli`, `kindling-sdk` → `spark-kindling-sdk`.
- PyPI collision: there is an unrelated, abandoned `kindling` package on PyPI (last release 2018, pytorch-domain). We chose a differentiated name rather than block on a PEP 541 transfer.
- The Python import name stays `kindling`; the `kindling` console script name is unchanged. No source-code migration is required for downstream consumers.

### One combined wheel with platform extras

- The three per-platform wheels (`kindling_synapse`, `kindling_databricks`, `kindling_fabric`) are replaced by a single `spark_kindling-<version>-py3-none-any.whl`.
- Platform-exclusive runtime deps are declared as extras:
  ```bash
  pip install 'spark-kindling[synapse]'
  pip install 'spark-kindling[databricks]'
  pip install 'spark-kindling[fabric]'
  pip install 'spark-kindling[standalone]'
  ```
- `scripts/generate_platform_config.py` and the per-wheel file-stripping logic in `scripts/build.py` are removed.

### Dynamic platform loader via entry points

- `[tool.poetry.plugins."spark_kindling.platforms"]` advertises each platform's module.
- `initialize_platform_services` in `packages/kindling/bootstrap.py` now discovers platforms via `importlib.metadata.entry_points`, falls back to the legacy `kindling.platform_{name}` convention, and raises an actionable `RuntimeError` that points the user at `pip install 'spark-kindling[<platform>]'` when the matching extra is missing.
- Third-party distributions can register additional platforms by contributing an entry point in the same group.

### Runtime bootstrap

- `runtime/scripts/kindling_bootstrap.py` now installs `spark_kindling-*.whl[<detected_platform>]` from artifacts storage. It keeps a legacy fallback to `kindling_{platform}-*.whl` for environments that still have pre-rename artifacts staged.

### Lazy imports in platform_synapse

- Removed two unused module-level imports (`ArtifactsClient`, `from azure.synapse.artifacts.models import *`) from `packages/kindling/platform_synapse.py`. The module can now be imported in any environment; Synapse-exclusive deps only load when a `SynapseService` instance is actually constructed.

### New unit tests

- `tests/unit/test_platform_entry_points.py` — asserts every expected platform is advertised via entry points, every entry point resolves to an importable module, and the install-with-extras error message fires on missing-module / factory-ImportError / unknown-platform paths.

## Validation

- Unit tests: `poetry run pytest tests/unit/ --no-cov -q` — 867 pre-existing tests pass plus 15 new ones.
- Real-cluster system tests run across all three platforms in parallel (build → deploy to Azure Storage → run):
  - **Fabric:** 16 passed, 1 skipped, 0 failed (18m)
  - **Databricks:** 14 passed, 3 failed (27m) — failures are timing/infra (`job timed out at 600s`, missing stdout markers, driver logs not found on shared cluster). No refactor-related failure signatures (no pip-install errors, no missing-extras errors, no platform-loader errors).
  - **Synapse:** 14 passed, 1 skipped, 2 failed (46m) — failures are timing (`job timed out at 600s`, `Unexpected final status: INPROGRESS`). No refactor-related failure signatures.

## Migration Notes

### Consumers installing from a release

Before:
```txt
kindling-synapse @ https://.../kindling_synapse-<version>-py3-none-any.whl
```

After:
```txt
spark-kindling[synapse] @ https://.../spark_kindling-<version>-py3-none-any.whl
```

### Consumers using the bootstrap from datalake

No action required. The bootstrap installs the new wheel with extras automatically based on `detect_platform()`. Legacy per-platform wheels staged from pre-rename deployments still resolve via the fallback path.

### Source-code

No changes. `import kindling.*` continues to work.

## Notes

- The two `test_default_telemetry` timeouts on Databricks and Synapse are the most noteworthy failures. A plausible contributor is that pip now resolves extras at job-install time against a slightly larger wheel; if these tests were historically close to the 600s budget, they may need a timeout bump or a pre-provisioned install step to stay green on those platforms.
- Follow-ups not in this release: renaming the `kindling-otel-azure` / `kindling-databricks-dlt` extension distributions; renaming the `kindling` console-script binary. Both carry user-facing breakage that's worth a dedicated release.
