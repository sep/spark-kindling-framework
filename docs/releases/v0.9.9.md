# Release Notes - v0.9.9

## Runner-Aligned Remote App Execution

Remote `kindling app run` now submits work through the durable Kindling runner instead of creating a one-off platform job definition per app.

### What changed

**Before:** `kindling app run <app> --platform <platform>` created a platform job definition for the app and started it directly. Each run could create or update an app-specific job definition in the workspace.

**After:** `kindling app run <app> --platform <platform>` verifies the runner is installed, then submits an app-run request to it. The runner is the single execution vehicle; the platform job definition is its infrastructure, not a per-app artifact.

`app status`, `app logs`, and `app cancel` now use runner-aware SDK methods internally. The `app logs` command no longer requires `--job-id` for streaming.

### Migration

No action required for most users. The external command interface is unchanged:

```bash
kindling app run orders --platform synapse
kindling app status <run-id> --platform synapse
kindling app logs <run-id> --platform synapse
kindling app cancel <run-id> --platform synapse
```

**If you do not have a runner installed**, remote `app run` will now fail with a clear message:

```
No Kindling runner found on synapse.
Run 'kindling runner ensure --platform synapse' to install it.
```

Run `kindling runner ensure --platform <platform>` once per workspace before running apps remotely.

**If your automation assumed `app run` creates or updates app-specific platform jobs**, those jobs will no longer be created. Use `kindling runner ensure` for runner lifecycle and `kindling app run/status/logs/cancel` for app execution.

### New SDK surface

`PlatformAPI` now exposes runner lifecycle and runner-aligned app submission methods available to all platform adapters:

```python
api.ensure_runner(platform)
api.get_runner_status(platform)
api.repair_runner(platform)
api.delete_runner(platform)
api.submit_app_run(app_name, environment=None, parameters=None)
api.get_app_run_status(run_id)
api.get_app_run_logs(run_id, from_line=0, size=1000)
api.stream_app_run_logs(run_id, callback=..., poll_interval=5.0, max_wait=300.0)
api.cancel_app_run(run_id)
```

The lower-level SDK job primitives (`create_job`, `run_job`, `get_job_status`, etc.) remain available for platform internals and specialized automation.
