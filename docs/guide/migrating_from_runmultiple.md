# Migrating from Notebook-Based Execution (`runMultiple`)

## Who This Guide Is For

You have a set of Synapse or Fabric notebooks orchestrated with
`mssparkutils.notebook.runMultiple` (Synapse) or
`notebookutils.notebook.runMultiple` (Fabric) — a hand-maintained DAG of
notebook activities, each reading and writing lake tables — and you want to
move that pipeline onto Kindling's declarative pipe model.

This guide maps each `runMultiple` concept to its Kindling equivalent, walks
through the migration step by step, and calls out the behavioral differences
you need to plan for.

## The Core Mental Shift

With `runMultiple`, **you** declare the dependency edges between notebooks:

```python
dag = {
    "activities": [
        {"name": "bronze_orders", "path": "nb_bronze_orders", "dependencies": []},
        {"name": "silver_orders", "path": "nb_silver_orders", "dependencies": ["bronze_orders"]},
    ],
    "concurrency": 2,
}
mssparkutils.notebook.runMultiple(dag)  # notebookutils on Fabric
```

The DAG dict is a parallel structure you maintain by hand: when a notebook
starts reading a new table, nothing forces you to update its `dependencies`
list, and a stale edge silently produces wrong-order execution.

In Kindling, dependencies are declared on **data, not activities**. Each pipe
states which entities it reads (`input_entity_ids`) and which entity it writes
(`output_entity_id`). The `ExecutionOrchestrator` derives the execution DAG
from those declarations and runs pipes in topological *generations* — groups
of pipes with no dependencies on each other. There is no separate dependency
list to keep in sync; the ordering is always consistent with what the code
actually reads and writes.

## Concept Mapping

| `runMultiple` concept | Kindling equivalent |
|---|---|
| Notebook activity | `@DataPipes.pipe()` function |
| Lake table read/written by a notebook | `@DataEntities.entity()` definition |
| `dependencies: [...]` on an activity | Derived automatically from `input_entity_ids` / `output_entity_id` |
| The DAG dict itself | Generated by `ExecutionPlanGenerator` — never hand-written |
| `concurrency: N` | `kindling.execution.parallel: true` + `max_workers: N` config |
| `timeoutPerCellInSeconds` | `kindling.execution.pipe_timeout` config (per-pipe, in seconds) |
| `timeoutInSeconds` (whole DAG) | No direct equivalent — see [Behavioral Differences](#behavioral-differences) |
| `retry` / `retryIntervalInSeconds` | No built-in retry — see [Behavioral Differences](#behavioral-differences) |
| `args` / notebook parameters | Hierarchical configuration (`ConfigService`) |
| `mssparkutils.notebook.exit(value)` | Return a DataFrame; results flow through entities |
| `%run` shared-code notebooks | Ordinary Python modules in your package |
| Pipeline run status / snapshot UI | `ExecutionResult` + orchestrator signals |

## Step-by-Step Migration

### 1. Inventory your notebooks' inputs and outputs

For each notebook activity in your DAG, list the tables it reads and the
table it writes. This inventory *is* your future pipe registry — and it often
surfaces stale or missing `dependencies` entries in the existing DAG dict.

A typical `runMultiple` notebook has three parts:

```python
# nb_silver_orders (before)
orders = spark.read.format("delta").load("abfss://.../bronze/orders")      # read
silver = orders.filter(col("status") != "cancelled").dropDuplicates(["order_id"])  # transform
silver.write.format("delta").mode("overwrite").save("abfss://.../silver/orders")   # write
```

Only the middle part survives the migration. Reads and writes become the
framework's job.

### 2. Define entities for every table

Each table in the inventory becomes an entity. Entities carry the schema,
merge keys, and partitioning that were previously scattered across notebook
read/write cells:

```python
from kindling.data_entities import *

@DataEntities.entity(
    entityid="bronze.orders",
    name="Bronze Orders",
    merge_columns=["order_id"],
    tags={"layer": "bronze", "domain": "sales"},
    schema=orders_schema,
)

@DataEntities.entity(
    entityid="silver.orders",
    name="Silver Orders",
    merge_columns=["order_id"],
    tags={"layer": "silver", "domain": "sales"},
    schema=orders_schema,
)
```

See [Data Entities](./data_entities.md) for storage configuration, providers,
and path resolution.

### 3. Convert each notebook body into a pipe

Strip the read and write cells; keep the transformation. Inputs arrive as
DataFrame parameters (entity IDs with dots become underscores), and the
return value is persisted for you:

```python
from kindling.data_pipes import *

@DataPipes.pipe(
    pipeid="silver_orders",
    name="Silver Orders",
    tags={"layer": "silver"},
    input_entity_ids=["bronze.orders"],
    output_entity_id="silver.orders",
    output_type="table",
)
def silver_orders(bronze_orders):
    return bronze_orders.filter(col("status") != "cancelled") \
                        .dropDuplicates(["order_id"])
```

Notes for common notebook patterns:

- **`%run nb_helpers`** — move shared functions into a module in your
  package and import them normally.
- **Notebook parameters / widgets** — read them from configuration instead
  (see [Hierarchical Configuration](../contributing/platform_workspace_config.md)).
- **`mssparkutils.notebook.exit("...")`** — delete it. Data flows through
  entities; execution status flows through `ExecutionResult` and signals.
- **Incremental loads** — if the notebook tracked its own high-water mark,
  consider `use_watermark=True` on the pipe instead
  (see [Watermarking](../contributing/watermarking.md)).

### 4. Replace the `runMultiple` call

Where you previously built the DAG dict and called `runMultiple`, execute the
pipes through the DAG orchestrator:

```python
import kindling
from kindling.data_pipes import DataPipesExecution
from kindling.injection import get_kindling_service

kindling.initialize(app_name="sales_pipeline")

executer = get_kindling_service(DataPipesExecution)
executer.run_datapipes(
    ["bronze_orders", "bronze_customers", "silver_orders", "gold_order_summary"],
    use_dag=True,
)
```

Concurrency and error handling do not live in this call — they come from
configuration (see the next step). This is a deliberate improvement over
`runMultiple`, where the `concurrency` knob was buried in an orchestration
notebook; in Kindling it's deployable, per-environment config.

The order of the list does not matter — the orchestrator computes the correct
order from entity dependencies. Or use the orchestrator directly when you
want the result object:

```python
from kindling.execution_orchestrator import ExecutionOrchestrator
from kindling.injection import GlobalInjector

orchestrator = GlobalInjector.get(ExecutionOrchestrator)
result = orchestrator.execute_batch(pipe_ids)  # options resolve from config

print(result.get_summary())
# {'run_id': ..., 'generations_executed': 3, 'success_count': 4,
#  'failed_count': 0, 'all_succeeded': True, 'failed_pipes': [], ...}
```

### 5. Map execution options

Execution options are **config-first**: the values that lived in your
`runMultiple` DAG dict move into hierarchical configuration under
`kindling.execution.*`, where they can differ per platform, workspace, or
environment without code changes.

| Before (`runMultiple` DAG dict) | After (`kindling.execution.*` config) |
|---|---|
| `"concurrency": 4` | `parallel: true`, `max_workers: 4` |
| `"timeoutPerCellInSeconds": 900` | `pipe_timeout: 900` (per pipe, not per cell) |
| Activity fails → dependents skipped | `error_strategy: skip_dependents` — exact same semantics: only transitive consumers skip, independent branches keep running |
| Run everything, report failures at the end | `error_strategy: continue` |

```yaml
# settings.yaml (or a platform/workspace/environment layer)
kindling:
  execution:
    parallel: true
    max_workers: 4
    pipe_timeout: 900
    error_strategy: fail_fast
```

For spot-testing, every option can still be passed as a parameter to
`run_datapipes()` / `execute_batch()` — a passed value overrides config for
that run only:

```python
from kindling.generation_executor import ErrorStrategy

executer.run_datapipes(
    pipe_ids,
    use_dag=True,
    parallel=False,                            # just this run, e.g. while debugging
    error_strategy=ErrorStrategy.CONTINUE,
)
```

### 6. Verify the derived DAG

Before deleting the old DAG dict, confirm the derived plan matches your
expectations. `ExecutionPlanGenerator` — the same service the orchestrator
uses internally — generates the plan without executing anything:

```python
from kindling.execution_strategy import ExecutionPlanGenerator
from kindling.injection import GlobalInjector

plan_generator = GlobalInjector.get(ExecutionPlanGenerator)
plan = plan_generator.generate_batch_plan(pipe_ids)

for gen in plan.generations:
    print(f"Generation {gen.number}: {gen.pipe_ids}")
print(plan.get_summary())
# {'strategy': 'batch', 'total_pipes': 4, 'total_generations': 3,
#  'max_parallelism': 2, 'generation_sizes': [2, 1, 1], ...}
```

Each generation should correspond to a "level" of your old `runMultiple`
DAG — pipes in the same generation are the ones that ran under the same
`dependencies` prerequisites. `plan.find_pipe_generation("gold_summary")`
answers "when does this pipe run?" for spot checks. At runtime, the same
information is available via the `orchestrator.plan_generated` signal and
on each `GenerationResult` in `result.generation_results`.

## Before / After: A Complete Example

**Before** — four notebooks and a hand-maintained DAG:

```python
dag = {
    "activities": [
        {"name": "bronze_orders",    "path": "nb_bronze_orders",    "dependencies": []},
        {"name": "bronze_customers", "path": "nb_bronze_customers", "dependencies": []},
        {"name": "silver_orders",    "path": "nb_silver_orders",    "dependencies": ["bronze_orders"]},
        {"name": "gold_summary",     "path": "nb_gold_summary",
         "dependencies": ["silver_orders", "bronze_customers"]},
    ],
    "concurrency": 2,
    "timeoutPerCellInSeconds": 900,
}
notebookutils.notebook.runMultiple(dag)
```

**After** — pipes declare what they read and write; the DAG is derived:

```python
@DataPipes.pipe(pipeid="bronze_orders", name="Bronze Orders", tags={"layer": "bronze"},
                input_entity_ids=["landing.orders"], output_entity_id="bronze.orders",
                output_type="table")
def bronze_orders(landing_orders):
    return standardize_columns(landing_orders)

@DataPipes.pipe(pipeid="bronze_customers", name="Bronze Customers", tags={"layer": "bronze"},
                input_entity_ids=["landing.customers"], output_entity_id="bronze.customers",
                output_type="table")
def bronze_customers(landing_customers):
    return standardize_columns(landing_customers)

@DataPipes.pipe(pipeid="silver_orders", name="Silver Orders", tags={"layer": "silver"},
                input_entity_ids=["bronze.orders"], output_entity_id="silver.orders",
                output_type="table")
def silver_orders(bronze_orders):
    return bronze_orders.filter(col("status") != "cancelled")

@DataPipes.pipe(pipeid="gold_summary", name="Gold Order Summary", tags={"layer": "gold"},
                input_entity_ids=["silver.orders", "bronze.customers"],
                output_entity_id="gold.order_summary", output_type="table")
def gold_summary(silver_orders, bronze_customers):
    return silver_orders.join(bronze_customers, "customer_id") \
                        .groupBy("customer_id").agg(sum("amount").alias("total"))

result = orchestrator.execute_batch(
    ["bronze_orders", "bronze_customers", "silver_orders", "gold_summary"],
)
```

with the old DAG dict's knobs as configuration:

```yaml
kindling:
  execution:
    parallel: true      # was "concurrency": 2
    max_workers: 2
    pipe_timeout: 900   # was "timeoutPerCellInSeconds": 900
```

The orchestrator derives the same shape the DAG dict encoded by hand:

```
Generation 0: bronze_orders, bronze_customers   (run in parallel)
Generation 1: silver_orders
Generation 2: gold_summary
```

Note the subtle improvement: in the `runMultiple` version, `gold_summary`
waits for `silver_orders` *and* `bronze_customers` only because someone
remembered to write both edges. In Kindling, adding a new input to
`gold_summary` automatically adds the dependency.

## Behavioral Differences

**Parallelism is within a generation, and it's threads, not sessions.**
`runMultiple` runs notebook activities concurrently in the shared Spark
session; Kindling's `parallel` option similarly runs the pipes of one
generation on a driver-side `ThreadPoolExecutor` (capped at `max_workers`)
against the same Spark session. Generations themselves always run
sequentially. Practical consequence in both worlds: concurrency buys you
overlap in job scheduling, not extra cluster capacity.

**No built-in per-activity retry.** `runMultiple`'s `retry` /
`retryIntervalInSeconds` have no direct equivalent. Options: implement retry
inside the pipe function for known-transient operations, or run with
`ErrorStrategy.CONTINUE`, collect `result.failed_pipes`, and re-execute just
those (the DAG orchestrator will re-derive the correct sub-order).

**No whole-run timeout.** `timeoutInSeconds` for the entire DAG has no
equivalent; `pipe_timeout` applies per pipe. If you need a global budget,
enforce it in the caller (e.g., the platform job's timeout setting).

**Failure semantics.** `error_strategy: skip_dependents` reproduces
`runMultiple`'s behavior exactly: a failed pipe's transitive consumers are
skipped (reported with `reason: upstream_failed`) while independent branches
keep running. The default `FAIL_FAST` is stricter — a failed generation
stops the whole run — and `CONTINUE` is looser, running everything and
reporting failures at the end.

**Skipped pipes.** A pipe is skipped (not failed) when its first input
entity returns no data. This replaces the common notebook pattern of exiting
early when a source table is empty; skips are reported separately in
`ExecutionResult.skipped_count`.

**Cancellation is cooperative.** In parallel mode, `FAIL_FAST` cancels pipes
that haven't started yet, but a pipe already running in a thread runs to
completion — same as an in-flight notebook activity under `runMultiple`.

## Observability

Where you previously watched the `runMultiple` snapshot UI or parsed its
return dict, subscribe to orchestrator signals:

```python
from kindling.signaling import DataSignals

@DataSignals.handler("orchestrator.pipe_failed")
def on_pipe_failed(sender, pipe_id, error, **kwargs):
    log.error(f"Pipe {pipe_id} failed: {error}")
```

The executor emits `orchestrator.execution_started/completed/failed`,
`orchestrator.generation_started/completed`, and
`orchestrator.pipe_started/completed/failed/skipped` — enough to reconstruct
the same progress view, and they feed the tracing/logging providers
automatically (see [Logging & Tracing](../contributing/logging_tracing.md)).

## Incremental Adoption

You do not have to migrate the whole DAG at once:

1. **Keep the orchestrating notebook.** Replace its `runMultiple` call with
   `run_datapipes(..., use_dag=True)` and migrate leaf notebooks into pipes
   first. Kindling runs fine inside a Synapse or Fabric notebook.
2. **Migrate one branch at a time.** A `runMultiple` activity can call a
   notebook that executes a Kindling sub-DAG, so old and new can coexist
   while you convert branch by branch.
3. **Retire the notebook shell last.** Once everything is pipes, package the
   pipeline as a data app and run it as a Spark job instead of a scheduled
   notebook — see [Job Deployment](../contributing/job_deployment.md).
